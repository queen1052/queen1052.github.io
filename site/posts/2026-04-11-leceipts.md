## 소개: AI는 왜 "완료했습니다"라고 거짓말을 할까

AI 코딩 어시스턴트를 쓰다 보면 누구나 한 번씩 이 경험을 한다.

> **당신**: "로그인 버그를 고쳐줘"
>
> **AI**: "완료했습니다! 로그인 버그를 수정했습니다. 모든 게 정상 작동합니다."
>
> *(실행해보니 여전히 크래시)*

AI가 악의적으로 거짓말하는 게 아니다. 검증이 일어났다는 것을 증명할 구조적 슬롯이 없기 때문이다. "정직하게 해줘"라고 말해도 소용없다 — AI는 yes라고 말하고 계속 거짓말한다. 하지만 검증 섹션이 필수인 응답 포맷을 강제하면, 빈 검증 박스는 명백히 눈에 보인다.

인간도 마찬가지다. "봤어요, 배포합니다"라고 말하는 PR에서 실제로 테스트를 안 하는 상황이 많다.

**leceipts**(`/rɪˈsiːts/`, "영수증"으로 발음)는 이 문제를 하나의 메커니즘으로 해결한다: 모든 코드 변경이 준수해야 하는 **5섹션 응답 포맷** + 영속화된 보고서 파일에 구조를 강제하는 **독립 체커**. GitHub 스타 26개, TypeScript 100%, MIT 라이선스.

## 주요 기능

**5섹션 포맷**: 루트 원인→변경 내용→재발 방지→검증→남은 위험. 이 구조 자체가 품질 게이트다. 섹션 4(검증)는 실제 명령어와 결과를 포함해야 한다.

**언어 불가지론적**: TypeScript, Python, Rust, Go — 어느 프로젝트에서도 사용 가능하다. 체커 스크립트만 Node.js 18+가 필요하고, 보고서는 순수 마크다운이다.

**AI 선택적**: AI 코딩 에이전트를 위해 만들었지만, 순수 인간 팀에서도 동일하게 작동한다.

**제로 런타임 의존성**: 체커에는 tsx와 TypeScript만 필요하다. 프로덕션 번들에 아무것도 추가되지 않는다.

**CI 통합**: GitHub Actions, pre-commit 훅, 빌드 스크립트 모두에 쉽게 추가할 수 있다.

## 5섹션 포맷 상세 분석

```
1. 루트 원인 (Root Cause)   — 실제로 뭐가 잘못됐고 왜인지
2. 변경 내용 (Change)       — 어떤 파일/라인이 수정됐는지
3. 재발 방지 (Recurrence prevention) — 이 버그가 다시 조용히 발생할 수 없는 이유
4. 검증 (Verification)      — 실행한 명령어 + 실제 결과
5. 남은 위험 (Remaining risk) — 회귀 위험과 미검증 영역 (위시리스트 아님)
```

각 섹션이 존재하는 이유:

| 섹션 | 막는 실패 |
|------|----------|
| 1. 루트 원인 | "고쳤더니 작동해요" — 이유를 모르면 증상 패치일 가능성 높음 |
| 2. 변경 내용 | 조용한 범위 확장 — 뭘 건드렸는지 명시적 목록 강제 |
| 3. 재발 방지 | 임시방편을 진짜 수정으로 포장 — 가드레일, 테스트, 또는 명시적 "의도적 생략" 강제 |
| 4. 검증 | 핵심 거짓말: "테스트했어요" 아무 증거 없이 — 명령어+결과 강제 |
| 5. 남은 위험 | 과신 — 테스트 안 한 것에 대한 정직한 공개 강제 |

섹션 5에는 역직관적인 규칙이 있다: **위시리스트가 아니다**. "더 강화할 수 있었는데", "후속 개선으로" 같은 문구가 있으면 체커가 위반으로 표시한다. 미검증 영역과 회귀 위험만 적어야 한다.

## 체커의 두 패스

### Pass 1: 구조적 검증

5개 섹션 모두가 존재하는지 확인한다. 섹션 제목은 한국어, 영어 모두 허용된다.

```
[leceipts] FAIL — 1 report(s) have violations:

  plans/TASK-042-verification-report.md
    - 3. Recurrence prevention / 재발 방지
    - 5. Remaining risk / 남은 리스크
```

### Pass 2: 내용 품질 검증

- **섹션 3**: 본문이 비어있거나 플레이스홀더만 있으면 위반
- **섹션 4**: 비어있거나 검증 증거(백틱 명령어, ✅/❌ 결과 마커, "검증 불가" 명시)가 없으면 위반
- **섹션 5**: "follow-up", "nice to have", "후속 개선" 같은 위시리스트 표현이 있으면 위반

```
[leceipts] FAIL — 1 report(s) have violations:

  plans/TASK-042-verification-report.md
    - 3. Recurrence prevention — empty or placeholder-only body
    - 4. Verification — no command, result marker, or 'unverifiable' statement found
    - 5. Remaining risk — wishlist phrasing detected: follow-up, nice to have
```

## 아키텍처: 3레이어 시스템

```
1. 프롬프트 레이어 (docs/working-rules.md)
   → CLAUDE.md / AGENTS.md / .cursorrules에 붙여넣기
   → AI가 포맷을 자동 사용

2. 아티팩트 레이어 (templates/verification-report-template.md)
   → plans/<ticket-id>-verification-report.md에 영속화
   → 검색 가능, 감사 가능

3. 강제 레이어 (scripts/check-reports.ts)
   → CI에서 실행, 위반 시 병합 차단
   → 부정행위 불가능
```

3가지 레이어 중 일부만 채택해도 된다. 프롬프트 레이어만 추가해도 즉시 응답 품질이 향상된다.

## 설치와 사용법

```bash
npm install --save-dev leceipts tsx
```

### 1. 설정 파일 생성

```json
// verification-kit.config.json
{
  "reportsDir": "plans",
  "reportPattern": "-verification-report\\.md$",
  "legacyMarker": "<!-- legacy-verification-report: pre-5-section format -->",
  "baseBranch": "main"
}
```

### 2. 템플릿 복사

```bash
mkdir -p plans
cp node_modules/leceipts/templates/verification-report-template.md \
   plans/verification-report-template.md
```

### 3. CLAUDE.md에 규칙 추가

`docs/claude-md-snippet.md`를 읽고 해당 스니펫을 `CLAUDE.md` / `AGENTS.md` / `.cursorrules`에 붙여넣는다.

### 4. 체커 실행

```bash
npx tsx node_modules/leceipts/scripts/check-reports.ts \
  --config ./verification-kit.config.json
```

### 5. npm 스크립트 추가

```json
{
  "scripts": {
    "check:reports": "tsx node_modules/leceipts/scripts/check-reports.ts --config ./verification-kit.config.json"
  }
}
```

### 6. GitHub Actions 통합

```yaml
- name: Check verification reports
  run: npm run check:reports
```

### 체커 모드

```bash
npx tsx scripts/check-reports.ts              # 새 보고서만 검사 (기본)
npx tsx scripts/check-reports.ts --all        # 모든 보고서 검사
npx tsx scripts/check-reports.ts --file <path>  # 특정 파일만
```

## 마치며

leceipts는 26스타짜리 작은 프로젝트지만, 제기하는 문제는 크다. AI 코딩 에이전트가 일상이 된 개발 환경에서 "완료" 신호를 믿을 수 있느냐의 문제다.

체커 스크립트는 ~300줄, 아주 단순하다. 그러나 이 단순한 구조적 강제가 "내 컴퓨터에서는 됐어요"를 구조적으로 불가능하게 만든다. 검증 박스가 비어 있으면 명백히 보인다. AI도, 인간도, 빈 박스를 제출하기가 민망해진다.

모든 주장에는 영수증이 필요하다. leceipts가 그 영수증을 요구한다.
